## Bug #2 found and fixed: aux-storage lock race destroys a good install

**Symptom:** Right after `RestoreImageManager.install()` finished (100%),
`automateSetupAssistant`'s `let vm = try VirtualMachine(bundle: bundle)`
(`Create.swift`, was line 1144) threw a Virtualization.framework error:
`"Invalid virtual machine configuration. Failed to lock auxiliary storage."`
тАФ and because that line sat **outside** the function's own `do { } catch { }`
block (the one specifically built to capture Setup-Assistant failures and
preserve the bundle for `--github-runner`, per the extensive comments on
`RunnerSetupAutomationFailure`), the error propagated straight past that
protection to `run()`'s generic catch, which prints the error and calls
`try? FileManager.default.removeItem(at: bundleURL)` тАФ deleting a bundle
that had just finished a real ~9-minute macOS install. Reproduced empirically
on this run; not simulated.

**Root cause, two layers:**
1. **Timing:** `VZMacAuxiliaryStorage(contentsOf:)` (invoked inside
   `VirtualMachineConfiguration.applyPlatform`) takes an exclusive lock on
   `auxiliary.bin`. The just-finished installer's own `VZVirtualMachine`
   object can still hold that lock for a brief window after the Swift
   object goes out of scope тАФ Virtualization.framework tears down/releases
   file locks asynchronously on its own internal queue, not synchronously
   on ARC dealloc. Constructing a second `VZVirtualMachine` on the same
   bundle immediately after `install()` returns can lose that race. (No
   official Apple doc documents an exact settle time; this is an empirical,
   reproduced behavior of the framework, consistent with independent
   community reports of needing a multi-second wait after
   `VZMacOSInstaller.install()` before touching the same disk/aux-storage
   files again.)
2. **Architecture gap:** even setting the timing issue aside, VM
   construction (and `vm.start()`) lived outside the do/catch meant to
   protect exactly this phase, so *any* construction/boot failure тАФ not
   just this one тАФ would have deleted a good bundle, contradicting the
   function's own documented contract.

**Fix (`Sources/spooktacular-cli/Commands/Create.swift`,
`automateSetupAssistant`):**
- Moved VM construction and `vm.start()` inside the existing do/catch so
  any failure there is captured as `automationFailure` (preserving the
  bundle and reporting via `RunnerSetupAutomationFailure`), exactly like
  every other failure in this phase. `vm` is now `Optional`, set only on
  successful construction, and the trailing `vm.stop()` is now
  `if let vm { ... }` guarded (nothing to stop if construction never
  succeeded).
- Added `Create.makeVirtualMachineWithRetry(bundle:)`: a small
  `@MainActor` bounded retry (4 attempts, `2^(attempt-1)`-second backoff тАФ
  same shape as the existing `RestoreImageManager.downloadWithRetry`) around
  `VirtualMachine(bundle:)` construction. Retries unconditionally on any
  construction error rather than pattern-matching a specific `VZError`
  code, since no verified Apple documentation was available in this
  environment to confirm the exact error-code enum case for this failure
  (per the project's "cite Apple docs, no extrapolation" rule) тАФ an
  unconditional bounded retry is honest about what's actually verified
  (the empirical failure and its transience) without asserting an
  unconfirmed API detail. This is a poll of the real success condition
  (construction not throwing), not a blind `Task.sleep` race against an
  assumed completion time.
- Replaced the `guard let underlyingVM = vm.vzVM else { throw ExitCode.failure }`
  early-exit with a descriptive `NSError` so it flows through the same
  catch/report path with a useful message instead of a bare exit code.

`swift test --filter Runner` 102/102 pass; `./build-app.sh release`

## Bug found and fixed (small unblock fix, in-scope per task brief)

**Symptom:** `spook create --from-ipsw <local-cached-ipsw>` fails the
host/guest compatibility check even though the local IPSW (26.4.1) is fully
compatible with this host (26.4.1) тАФ because the CLI unconditionally called
`RestoreImageManager.fetchLatestSupported()` (which queries Apple's catalog
for the network "latest" тАФ 26.5.2 at time of run тАФ and check *that* version
against the host) before ever branching on `--from-ipsw`. So creating from a
cached, already-compatible local IPSW was completely broken whenever Apple's
network catalog had shipped a newer release than this host's OS.

**Where:** `Sources/spooktacular-cli/Commands/Create.swift`, in `Create.run()`
around the restore-image resolution block (was ~line 437-478).

**Confirms it's a real regression, not a design choice:** the exact same bug,
with the exact same fix, was already shipped for the GUI path in
`Sources/Spooktacular/AppState.swift` (`runMacOSCreate`, ~line 1004-1052),
whose comment documents this precise failure mode:

> "Previously this path unconditionally called `fetchLatestSupported()`
> before branching, which made every create (including local-IPSW creates)
> depend on Apple's catalog reachability тАФ a single 'restore image catalog
> failed to load' bubbled up from a network failure on the host blocked
> users who already had the IPSW on disk."

The CLI's `Create.swift` was never updated to match тАФ this task's live run
was the first thing to exercise `--from-ipsw` with a cached file against a
host older than the current Apple catalog "latest", surfacing the gap.

**Fix:** Mirrored the GUI's already-correct pattern into the CLI: branch on
`fromIpsw == "latest"` *before* touching the network. For `"latest"`, keep
the existing `fetchLatestSupported()` + `downloadIPSW()` flow unchanged. For
a local path, load restore-image metadata straight from disk via
`VZMacOSRestoreImage.image(from:)` (Apple's documented API for local IPSW
inspection, same API the GUI already uses) тАФ no network call, no
"latest"-vs-host compatibility gate. Also fixed a `macOSMajor` variable that
referenced the old unconditional `version` local тАФ derived it from
`restoreImage.operatingSystemVersion` instead, which is in scope from either
branch.

**Verification:** `swift build -c release --product spooktacular-cli` and a
full `swift build` (all targets) both pass. `./build-app.sh release` rebuilt
cleanly with the fix; the Provisioner pkg is still correctly nested. The
live `spook create runner-e2e --from-ipsw <cached file>` run below is itself
the empirical verification that the fix unblocks local-IPSW creation.

No dedicated unit test target covers `spooktacular-cli`'s `Create.run()`
today (no `Tests/` directory for that executable target тАФ it's exercised via
manual/e2e runs only, confirmed by grep). Did not stand up a new test
harness for this fix given the scope of this verification task; flagging
that gap for a follow-up task rather than expanding scope here.

(`9fea23b4e`, `06c7e0ba8`) prior to this attempt-2 session.

## Root cause: not determined тАФ flagging as a real bug, not environment noise

Both live runs failed identically at the same step with the same error.
This is a genuine gap in the `--github-runner` create flow found by this
verification task, distinct from (and downstream of) the two bugs already
fixed earlier in this effort (local-IPSW catalog rejection; aux-storage
lock). Recommended follow-up (out of scope for this verification task
per its own instructions тАФ diagnose-and-fix was not the mandate, proving
the flow works end-to-end was):

1. Instrument `IPResolver`/the Setup Assistant automation path to log
   guest-visible network state (or capture a screenshot via
   `VZScreenReader` at the moment of IP-resolution failure) so a future
   attempt doesn't have to infer guest-side state purely from host-side
   DHCP lease/ARP artifacts.
2. Investigate whether macOS 26's Setup Assistant/OOBE flow defers full
   network-interface bring-up behind a screen this automation sequence
   doesn't handle (network selection, "Not Now" on Wi-Fi, etc.) тАФ the
   symptom (Terminal-typed commands all "succeed" per the executor, but
   the guest never got a DHCP lease) is consistent with the automation
   reaching a Terminal window that isn't actually backed by a
   fully-initialized network stack yet.
3. Consider raising `IPResolver.resolveIPWithRetry`'s default timeout
   (120s) and/or lengthening the delay between "Setup Assistant
   automation complete" and the first IP-resolution poll, in case this
   is DHCP-lease latency rather than a hard failure тАФ though 8-10
   minutes of prior guest uptime in both runs makes a pure "needs more
   time" explanation unlikely on its own.

## DHCP root cause (found and fixed тАФ 2026-07-03 follow-up session)

**The "DHCP lease never appears" verdict above was a misdiagnosis тАФ the
lease was granted correctly in both runs. The bug is a MAC-address
string-format mismatch in `IPResolver`, not a networking failure.**

**Root cause:** `Sources/SpooktacularInfrastructureApple/IPResolver.swift`,
`parseLeases(_:macAddress:)` (was line 166) and `parseARPOutput(_:macAddress:)`
(was line 204) compared the VM's MAC тАФ always canonically zero-padded
two-digit hex octets via `SpooktacularCore/MACAddress.rawValue` тАФ against
raw text extracted from two host data sources that do **not** zero-pad:

- `/var/db/dhcpd_leases`'s `hw_address=`/`identifier=` fields, written by
  `bootpd`.
- `arp -an`'s `at <mac>` field.

Both format each octet with a bare `%x`, so any octet below `0x10` prints
as a single hex digit (`"1"` not `"01"`, `"a"` not `"0a"`). An exact-string
comparison (`mac == macAddress`) or substring search
(`line.contains(macAddress)`) against the fully-padded target therefore
silently fails whenever the VM's randomly generated MAC has **any** octet
below `0x10` тАФ roughly two-thirds of addresses produced by
`MACAddress.generate()`.

**Empirical proof, using the real data from this same host:**
- `arp -an` right now prints the well-known multicast MAC
  `01:00:5e:00:00:fb` as `1:0:5e:0:0:fb`.
- Re-reading `/var/db/dhcpd_leases` (never cleared across sessions) found
  the lease for attempt 1's exact MAC from the run above,
  `de:2a:2d:f3:01:b8`, recorded as `hw_address=1,de:2a:2d:f3:1:b8` тАФ
  `lease=0x6a476323` decodes to `2026-07-03 00:22:11`, which is exactly
  the VM's 1-hour default lease duration past its `23:22:1x` boot time
  logged above. The lease was granted essentially immediately at boot.
- Attempt 2's (retry) MAC likewise has a matching lease
  (`hw_address=1,12:9e:3:38:c6:f`, `lease=0x6a476827` тЖТ expiry
  `2026-07-03 00:43:35`, again ~1 hour past its `23:43:36` boot time).

So DHCP worked correctly both times; `IPResolver` (and the human eye
scanning the raw file for `de:2a:2d:f3:01:b8` during the original
diagnosis, which also wouldn't find `de:2a:2d:f3:1:b8`) simply couldn't
recognize the match.

**Empirical discriminator run (this follow-up session):** created a
throwaway VM (`nettest`, `--skip-setup --from-ipsw <cached 26.4.1>`),
booted it, and polled `spook ip nettest` тАФ resolved `192.168.64.97`
within about a minute of boot, confirmed against the raw lease record
(`hw_address=1,c2:ab:c1:20:b1:59` тАФ this MAC happens to have no octet
below `0x10`, so it "accidentally" worked even pre-fix, which is exactly
why the bug is intermittent-looking rather than always-broken).

**Fix:** Added `IPResolver.normalizeMACAddress(_:)`, which splits a
colon-separated MAC string, validates each component as 1-2 hex digits,
and zero-pads to a canonical two-digit lowercase form. Both
`parseLeases` and `parseARPOutput` now normalize the MAC text extracted
from their respective raw sources (and the search target) through this
helper before comparing, instead of comparing raw/padded strings
directly. `parseARPOutput` was also tightened to extract the MAC token
between `" at "` and `" on "` rather than doing a raw substring
`.contains(_:)` search.

**Tests:** `Tests/SpooktacularKitTests/IPResolverTests.swift` gained:
- A `MAC address normalization` suite unit-testing
  `normalizeMACAddress(_:)` directly (padding, case, and malformed-input
  rejection).
- Regression fixtures in both the ARP and lease suites built from the
  *exact* raw strings captured from this host (the `01:00:5e:00:00:fb`
  and `de:2a:2d:f3:01:b8`/`de:2a:2d:f3:1:b8` cases above), verified to
  **fail against the pre-fix source** (confirmed by temporarily
  `git stash`-ing just the source file and re-running тАФ all 5 new cases
  failed as expected) and pass with the fix restored.

**Verification:** `swift build` clean; `swift test` 810/810 pass;
`swiftlint --strict` 0 violations, 0 serious, across 289 files; live
`spook ip nettest` resolution succeeded post-fix as described above.

**Scope note:** this fix corrects IP resolution itself. It does not
re-run the full `--github-runner` end-to-end flow (registration-token
mint тЖТ runner-template injection тЖТ online poll) тАФ that remains the
next verification step, now unblocked at the layer this session
diagnosed. The `spook delete` MFA/headless-teardown gap noted above is
unrelated and still open.
