Versioning & releases

Spooktacular follows Semantic Versioning 2.0.0 with clearly defined pre-release stages and compatibility guarantees.

Version format

1Major
.
2Minor
.
3Patch

MAJOR — Breaking changes

Incremented when we make breaking changes to the public API, CLI interface, VM bundle format, or configuration schema. Enterprise teams pinning to a major version can upgrade minor and patch versions without risk.

What breaks: command syntax changes, config.json schema changes, removed commands, renamed flags, changed default behavior.

Promise: We will not ship a major version bump without a migration guide and at least one release candidate.

MINOR — New features

Incremented when we add new backward-compatible features. Existing VMs, scripts, and CI configurations continue to work unchanged.

Examples: new CLI commands, new provisioning modes, new network modes, GUI improvements, new templates.

PATCH — Bug fixes

Incremented for bug fixes and security patches. No new features, no behavior changes. Safe to upgrade immediately.

Examples: crash fixes, error message improvements, documentation corrections, dependency updates.

Pre-release stages

-alpha.N

Alpha

Early development. APIs may change between alphas. Not recommended for production. Useful for testing new features before they stabilize.

-beta.N

Beta

Feature-complete for the target release. APIs are stable but may have bugs. Suitable for staging environments and early adopter testing.

-rc.N

Release Candidate

Production-ready candidate. No known bugs. If no issues are found within the testing period, this becomes the stable release. Enterprise teams should test RCs in staging.

What CI checks on every PR:

  • swift test --parallel — all unit tests must pass
  • swift build -c release — release build must compile clean
  • .app bundle structure verification

What ships on every tag:

  • GitHub Release with .app.zip (signed + notarized)
  • TestFlight build (via Fastlane pilot)
  • App Store submission (via Fastlane deliver)
  • DocC documentation update (via GitHub Pages)

Release schedule

We do not follow a fixed calendar. Releases ship when they are ready, not on arbitrary deadlines.

  • Patch releases ship within 48 hours of a confirmed bug fix
  • Minor releases ship when a meaningful set of features is complete and tested (typically every 2–4 weeks during active development)
  • Major releases ship only when necessary (breaking changes), with at least 2 weeks of release candidate testing
  • Release candidates remain in testing for a minimum of 1 week. If issues are found, additional RCs ship until clean

VM bundle format

The .vm bundle directory structure (config.json, disk.img, platform artifacts) is versioned separately from the tool. If we change the bundle format, we will:

  • Support reading old bundles indefinitely
  • Provide automatic migration on first access
  • Document the changes in release notes

How to pin versions

Pin to a specific version 
# Homebrew
brew install --cask spooktacular@1.0

# GitHub Actions
- uses: spooky-labs/setup-spooktacular@v1

# Direct download (specific version)
curl -L https://github.com/Spooky-Labs/spooktacular/releases/download/v1.0.0/Spooktacular.app.zip

Reporting issues

If you find a bug in a release:

  • Check the release notes to see if it's already fixed in a newer version
  • Open an issue with your version number, macOS version, and steps to reproduce
  • Patch releases ship within 48 hours of confirmed fixes
