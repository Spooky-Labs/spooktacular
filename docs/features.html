⚙ For CI/CD Teams

Double your GitHub Actions capacity overnight

Every EC2 Mac or Mac mini becomes two self-hosted runners. Clone VMs in milliseconds, provision with user-data, replace on exit.

The problem

  • EC2 Mac dedicated hosts cost $26/day each, running one bare-metal runner
  • Golden images take hours to build and maintain for each Xcode version
  • Runners accumulate state between jobs, causing flaky builds
  • No native way to run 2 isolated CI environments on one Mac

The Spooktacular solution

  • Run 2 VMs per Mac — 50% cost reduction, same hardware
  • User-data scripts run via SSH provisioning; disk-inject mode in progress
  • Ephemeral runners: clone, use, delete, repeat — zero state leakage
  • APFS copy-on-write cloning: 48ms for a 30 GB VM image
setup-gha-runners.sh 
# Create a base VM with Xcode pre-installed
spook create base --from-ipsw latest --cpu 8 --memory 16 --disk 100

# Clone ephemeral runners (48ms each, shares disk blocks via APFS CoW)
spook clone base runner-01
spook clone base runner-02

# Start with GitHub Actions runner provisioning
spook start runner-01 --headless --user-data ./install-gha-runner.sh
spook start runner-02 --headless --user-data ./install-gha-runner.sh

# Or generate a GitHub runner install script from the built-in template
spook create runner --github-runner --github-repo myorg/myapp --github-token $TOKEN
# Then start with: spook start runner --provision ssh

Instant cloning Shipped

APFS copy-on-write duplicates a 30 GB VM in 48ms. Both VMs share physical disk blocks. Each clone gets a fresh MachineIdentifier.

SSH provisioning Shipped

SSH provisioning waits for IP, connects, and executes a user-data script with streaming output. Disk-inject provisioning is in progress. Guest agent (vsock) and shared-folder modes are planned.

Pool auto-replacement Planned

Runners automatically replaced on exit. GitHub webhook receiver for workflow_job events. Configurable min/max replicas and idle timeout.

🧠 For ML Engineers

Apple Silicon GPU in every VM

Metal GPU acceleration, shared folders for terabytes of training data, and host-only networking for federated learning.

The problem

  • ML training on Mac needs isolation without sacrificing GPU access
  • Copying terabytes of training data into each VM is impractical
  • Federated learning requires VM-to-VM communication without internet
  • Training runs crash — restarting from zero wastes GPU hours

The Spooktacular solution

  • Metal GPU via VirtIO — MLX, PyTorch MPS, and Core ML all work
  • VirtIO shared folders mount host datasets at native speed, no copying
  • Host-only networking: VMs communicate without external access
  • VM snapshots save training state, restore in seconds on crash
ml-training-setup.sh 
# Create a VM optimized for ML workloads
spook create ml-worker --from-ipsw latest --cpu 10 --memory 32 --disk 200

# Mount training data without copying (read-only for safety)
spook share ml-worker add /data/imagenet --tag imagenet --read-only
spook share ml-worker add /data/checkpoints --tag checkpoints

# Start with ML environment provisioning
spook start ml-worker --user-data ./install-mlx-pytorch.sh

# Save training checkpoint as VM snapshot
spook snapshot ml-worker epoch-50

Metal GPU displays Shipped

1-2 Metal GPU-accelerated displays per VM. Apple Silicon GPU is exposed via the Virtualization framework for rendering.

Shared folders Shipped

Mount host directories via VirtIO FS at /Volumes/My Shared Files/. Read-only or read-write per folder. No data duplication.

Host-only networking Shipped

VMs communicate with each other and the host but have zero internet access. Ideal for federated learning and secure environments.

🖨 For Remote Desktop

macOS desktops on demand

Spin up isolated macOS environments with VNC access. Auto-resize displays, dual monitor support, clipboard sharing.

The problem

  • macOS testing requires real desktops with real display output
  • Shared machines accumulate state and configuration conflicts
  • Remote desktop solutions require manual setup in each environment
  • Display resolution is fixed, doesn't adapt to client window

The Spooktacular solution

  • Remote desktop template generates a script to enable Screen Sharing (run via spook start --provision ssh)
  • Each developer gets a fresh, isolated macOS VM with full display
  • Auto-resize: guest resolution follows the display window (macOS 14+)
  • Audio, microphone, and clipboard sharing are shipped VM features
remote-desktop.sh 
# Create a desktop VM with dual displays
spook create desktop --from-ipsw latest --cpu 8 --memory 16 --displays 2

# Enable Screen Sharing via built-in template
spook start desktop --user-data ./enable-vnc.sh

# Get the VNC URL
spook ip desktop
192.168.64.3
# Connect via Screen Sharing: vnc://192.168.64.3

Auto-resize display Shipped

Guest resolution automatically follows the host window size on macOS 14+. No manual resolution configuration needed.

Audio + microphone Shipped

VirtIO sound with optional host microphone passthrough. Enable or disable per VM in the configuration.

4K 60fps HEVC Planned

Planned: hardware-accelerated encoding via VideoToolbox with adaptive bitrate. No code exists yet.

⎈ For DevOps

Kubernetes-native fleet management

Manage macOS VMs like pods. CRDs, operators, Helm charts, autoscaling. Use your existing K8s toolchain.

The problem

  • macOS VMs don't fit into Kubernetes-native infrastructure workflows
  • EC2 Mac fleet scaling requires custom tooling and manual intervention
  • No standard way to declare macOS runner pools as infrastructure-as-code
  • Monitoring and lifecycle management are ad-hoc per-host scripts

The Spooktacular solution

  • (Planned) MacOSVM and MacOSVMPool CRDs — declare VMs in YAML, apply with kubectl
  • (Shipped) 2-VM-per-Mac capacity enforcement via kernel limit detection
  • (Planned) Fleet scaling models: AWS ASG, K8s cluster-autoscaler, built-in
  • (Planned) Helm chart for installation, Terraform module for EC2 Mac provisioning
macos-runners.yaml 
apiVersion: spooktacular.dev/v1alpha1
kind: MacOSVMPool
metadata:
  name: ci-runners
spec:
  replicas: 4
  template:
    spec:
      image: ghcr.io/spooktacular/macos-xcode:15.4-16.2
      cpu: 8
      memory: 16Gi
      userData: ./setup-gha.sh
      ephemeral: true  # Delete and recreate after each job

Kubernetes CRDs Planned

MacOSVM and MacOSVMPool custom resources. Swift operator compiled to Linux ARM64. Helm chart for installation.

EC2 Mac one-liner Planned

Single curl command sets up Spooktacular on a fresh EC2 Mac with GitHub Actions runner, Xcode, and autoscaling.

CLI (16 commands) Shipped

create, start, stop, list, clone, delete, ip, get, set, snapshot, restore, snapshots, share, ssh, exec. Styled ANSI output, --json for automation.
