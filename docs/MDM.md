# Spooktacular Embedded MDM

Spooktacular ships an embedded MDM (mobile device management) server so
the host can run a single command — `spook mdm enroll my-vm` — and have
the VM enroll into a host-running MDM on its next boot. Once enrolled,
the host can push profiles, install pkgs, and run user-data scripts
without per-VM admin password prompts.

No third-party Jamf / Kandji needed for VM-scope management.

## What it solves

- **CI runners** — bootstrap N ephemeral VMs the same way every time.
- **Remote desktops** — push profiles + apps to managed dev VMs.
- **Multi-tenant** — different egress / profile policies per tenant
  without operator-per-tenant overhead.

## The flow

```sh
# 1. Once per host (idempotent)
spook mdm init

# 2. Run the MDM server (foreground; or wrap in launchd)
spook mdm serve --host 192.168.64.1

# 3. Bootstrap each VM (in a separate terminal)
spook mdm enroll my-vm --server 192.168.64.1
spook start my-vm
# … VM boots → first-boot.sh → mdmclient enrolls

# 4. Push commands as needed
spook mdm run my-vm ./setup.sh    # push a script
spook mdm devices                 # list enrolled VMs
```

The VM boots, runs the injected `first-boot.sh`, installs the
enrollment profile, and `mdmclient` enrolls into the server. From
that point on, `spook mdm run <vm> <script>` pushes a one-shot pkg
that runs the script as root inside the guest.

## Prerequisites

- **Spooktacular Provisioner.pkg installed in the guest.** This is the
  one-time setup the user does inside Guest Tools (System Settings →
  Login Items & Extensions, approve once). The provisioner runs
  `first-boot.sh` on every boot — that's what triggers MDM enrollment.

  See [Guest Tools provisioning](GUEST_TOOLS.md) for details.

## Defaults

- **Signed enrollment.** Each VM gets a unique identity certificate
  signed by a host-resident root CA (generated by `spook mdm init`,
  stored at `~/.spooktacular/mdm/`).
- **TLS on the server.** The server presents a cert from the same root
  CA, with SAN matching the host address.

Override either with `--unsigned` (enroll) and `--no-tls` (serve) for
loopback / dev. Both flags must be paired — a signed profile can't
dial an HTTP server.

## Health check

```sh
spook mdm doctor
```

Validates that the root CA + binary dependencies (`/usr/bin/openssl`)
are in place. Useful before automating in CI.

## Inspecting an enrollment

```sh
# Render the bootstrap script without modifying the bundle
spook mdm enroll my-vm --print-only

# Inspect the per-VM profile inside the bundle
cat ~/.spooktacular/vms/<uuid>.vm/provision/first-boot.sh
```

## How it works

```
Host                                 Guest
─────────────────────────────────────────────────────────────
spook mdm init        ──┐
                        │
                  ~/.spooktacular/mdm/root-ca.{pem,key}
                  generated (P-256, openssl)

spook mdm serve  ─────► EmbeddedMDMServer listens on host
                        - /mdm/checkin   (Authenticate/etc.)
                        - /mdm/server    (commands, responses)
                        - /mdm/manifest  (pkg + script delivery)
                        Server cert minted from root CA, SAN
                        covers --host + --tls-san list.

spook mdm enroll      ──┐
                        │
                  Per-VM identity cert minted (CN = VM UUID),
                  PKCS#12 bundle embedded in .mobileconfig.
                  Bootstrap bash dropped to
                  <bundle>/provision/first-boot.sh.

spook start my-vm  ─►                  VM boots
                                       └─► spook-provision-runner
                                           reads first-boot.sh,
                                           runs as root:
                                           - drops .mobileconfig
                                           - `profiles install`
                                           - mdmclient dials host

                                       mdmclient enrolls via TLS:
                                       - Authenticate
                                       - TokenUpdate
                                       - polls /mdm/server
```

## Reference

| Command | Purpose |
|---|---|
| `spook mdm init [--storage DIR]` | Generate the root CA. Idempotent. |
| `spook mdm serve [--host H] [--port P] [--no-tls] [--tls-san DNS]` | Run the MDM listener. |
| `spook mdm enroll <vm> [--server H] [--unsigned] [--no-tls] [--print-only]` | Inject MDM bootstrap into a VM bundle. |
| `spook mdm run <vm> <script>` | Push a user-data script to an enrolled VM (`-` for stdin). |
| `spook mdm devices` | List enrolled VMs. |
| `spook mdm doctor` | Health-check the host setup. |

## Architecture notes

- **In-memory state today.** Device store + command queue live in the
  running `serve` process. A host restart drops them; devices
  naturally re-Authenticate on their next boot so enrollment is
  self-healing, but queued-but-unsent commands are lost. A
  JSON-backed snapshotter is on the roadmap.
- **No APNs.** The server doesn't issue push notifications;
  `mdmclient` polls instead. Sufficient for host-controlled VMs.
- **No Apple Business Manager.** VMs aren't DEP-registered; the
  enrollment profile is installed via the in-bundle bootstrap rather
  than push.
