# Spooktacular Embedded MDM

Spooktacular ships an embedded MDM (mobile-device-management)
server so the host can run a single command —
`spook mdm enroll my-vm` — and have the VM enroll into a
host-running MDM on its next boot. Once enrolled, the host can
push profiles, install pkgs, and run user-data scripts without
per-VM admin-password prompts.

No third-party Jamf / Kandji needed for VM-scope management.

## What it solves

- **CI runners** — bootstrap N ephemeral VMs the same way every time.
- **Remote desktops** — push profiles + apps to managed dev VMs.
- **Multi-tenant** — different egress / profile policies per tenant
  without operator-per-tenant overhead.

## The flow

```sh
# 1. Once per host (idempotent — generates the root CA)
spook mdm init

# 2. Run the MDM server (foreground; or wrap in launchd)
spook mdm serve --host 192.168.64.1

# 3. Bootstrap each VM (separate terminal)
spook mdm enroll my-vm --server 192.168.64.1
spook start my-vm
# … VM boots → first-boot.sh → mdmclient enrolls

# 4. Push commands as needed (also a separate terminal)
spook mdm run my-vm ./setup.sh    # push a script
spook mdm devices                 # list enrolled VMs
```

The VM boots, runs the injected `first-boot.sh`, installs the
enrollment profile, and `mdmclient` enrolls into the server.
From that point on, `spook mdm run <vm> <script>` queues a
one-shot pkg that runs the script as root inside the guest.

## Prerequisites

- **Spooktacular Provisioner.pkg installed in the guest.** This
  is the one-time setup the user does inside Guest Tools
  (System Settings → Login Items & Extensions, approve once).
  The provisioner runs `first-boot.sh` on every boot — that's
  what triggers MDM enrollment.

  See [Guest Tools end-to-end verification](GUEST_TOOLS_E2E_VERIFICATION.md)
  for the user-visible setup.

## Defaults

- **Signed enrollment.** Each VM gets a unique identity
  certificate signed by a host-resident root CA (generated by
  `spook mdm init`, stored at `~/.spooktacular/mdm/`).
- **TLS on the server.** The server presents a cert from the
  same root CA, with SAN matching the host address.

Override either with `--unsigned` (enroll) and `--no-tls`
(serve) for loopback / dev. Both flags must be paired — a
signed profile can't dial an HTTP server.

## Health check

```sh
spook mdm doctor
```

Validates that the root CA + binary dependencies
(`/usr/bin/openssl`) are in place. Useful before automating
in CI.

## Inspecting an enrollment

```sh
# Render the bootstrap script without modifying the bundle
spook mdm enroll my-vm --print-only

# Inspect the per-VM profile dropped into the bundle
cat ~/.spooktacular/vms/<uuid>.vm/provision/first-boot.sh
```

## How it works

```
Host                                 Guest
─────────────────────────────────────────────────────────────
spook mdm init        ──┐
                        │
                  ~/.spooktacular/mdm/root-ca.{pem,key}
                  generated (P-256, openssl).

spook mdm serve  ─────► EmbeddedMDMServer listens on host
                        - /mdm/checkin     (Authenticate/etc.)
                        - /mdm/server      (commands, responses)
                        - /mdm/manifest/   (manifest delivery)
                        - /mdm/pkg/        (pkg byte delivery)
                        Server cert minted from root CA; SAN
                        covers --host + each --tls-san.

spook mdm enroll      ──┐
                        │
                  Per-VM identity cert minted (CN = VM UUID),
                  PKCS#12 bag embedded in .mobileconfig.
                  Bootstrap bash dropped to
                  <bundle>/provision/first-boot.sh.

spook start my-vm  ─►                  VM boots
                                       └─► spook-provision-runner
                                           reads first-boot.sh,
                                           runs as root:
                                           - drops .mobileconfig
                                           - `profiles install`
                                           - mdmclient dials host

                                       mdmclient enrolls via TLS:
                                       - Authenticate
                                       - TokenUpdate
                                       - polls /mdm/server

spook mdm run …    ─────► CLI A: writes JSON request to
                          ~/.spooktacular/mdm/state/outbox/<uuid>.json
                          (atomic file write).

(same process)         ◄──── CLI B (`serve`): drains outbox
                          every 2s — builds pkg, registers in
                          content store, enqueues
                          InstallEnterpriseApplication
                          targeting the device's UDID.

                                       mdmclient's next idle
                                       poll returns the command;
                                       fetches /mdm/manifest/<id>
                                       then /mdm/pkg/<id>;
                                       runs `installer`; ack's.
```

## Reference

| Command | Purpose |
|---|---|
| `spook mdm init [--storage DIR]` | Generate the root CA. Idempotent. |
| `spook mdm serve [--host H] [--port P] [--no-tls] [--tls-san DNS]` | Run the MDM listener. |
| `spook mdm enroll <vm> [--server H] [--unsigned] [--no-tls] [--print-only]` | Inject MDM bootstrap into a VM bundle. |
| `spook mdm run <vm> <script>` | Push a user-data script to an enrolled VM (`-` for stdin). |
| `spook mdm devices` | List enrolled VMs (reads `state/devices.json`). |
| `spook mdm doctor` | Health-check the host setup. |

## Architecture map

### Layering

Clean architecture: each module imports only from layers
beneath it.

```
spook  CLI            ⟵ user-facing surface (ArgumentParser)
   │
   ▼
SpooktacularInfrastructureApple
   │   • EmbeddedMDMServer            Network.framework listener + TLS / mTLS
   │   • MDMIdentityIssuer            openssl-backed CA + per-VM cert minting
   │   • MDMUserDataPkgBuilder        pkgbuild + productbuild
   │
   ▼
SpooktacularApplication
   │   • SpooktacularMDMHandler       MDMServerHandler implementation
   │   • MDMUserDataDispatcher        "dispatch script to UDID" use case
   │   • MDMEnrollmentBootstrap       first-boot.sh renderer
   │   • MDMDispatchOutbox            file-backed IPC for `spook mdm run`
   │   • MDMDeviceStore               in-memory directory of enrolled devices
   │   • MDMDeviceStorePersister      JSON snapshotter for the device store
   │   • MDMCommandQueue              per-UDID FIFO + in-flight slot
   │   • MDMCommandResponse           ServerCommandResponse decoder
   │   • MDMCheckInMessage            check-in plist decoder
   │   • MDMContentStore              keyed cache of pkg + manifest bytes
   │   • MDMCommand                   command DTO + wire-format renderer
   │   • MDMManifestBuilder           Apple-format manifest plist
   │   • MDMServerHandler             transport-policy boundary
   │
   ▼
SpooktacularCore  (Foundation only)
```

### State on disk

```
~/.spooktacular/mdm/
  root-ca.pem               Self-signed P-256 root CA.
  root-ca.key               Matching EC private key, mode 0600.
  root-ca.srl               openssl serial-number tracking.
  state/
    devices.json            MDMDeviceStorePersister snapshot —
                            UDID, topic, model, OS version,
                            push-token metadata, firstSeen /
                            lastSeen, checkedOut flag. Rewritten
                            on every check-in / command response.
    outbox/
      <uuid>.json           One file per `spook mdm run`
                            invocation; deleted on successful
                            dispatch, renamed `<uuid>.failed.json`
                            on permanent failure.
```

### What's in-memory only

- **Command queue** (`MDMCommandQueue`) — per-UDID pending
  list + in-flight slot. A host restart re-Idles the device;
  any queued-but-unsent commands are lost. Devices retrying
  via `mdmclient`'s natural poll loop heal the case where
  nothing was in flight.
- **Content store** (`MDMContentStore`) — pkg + manifest
  bytes keyed by UUID, served via `/mdm/manifest/<id>` +
  `/mdm/pkg/<id>`. Lost on restart; the operator just
  re-`run`s.

The device-store snapshot is enough to make `spook mdm
devices` deterministic across `serve` restarts.

## Security posture

- **TLS by default.** `spook mdm serve` mints a server
  certificate signed by the root CA and presents it on every
  connection. SANs cover `--host` + each `--tls-san`. To
  disable, pass `--no-tls` (pair with `--unsigned` on enroll).
- **mTLS optional.** Pass `clientCAAnchorDER` to
  `EmbeddedMDMServer.init(...)` (programmatic surface) to
  require each connecting client to present a cert chaining to
  the host's root CA. The `ClientCertVerifier` pins the
  anchor and refuses system roots, so a globally-trusted CA
  can't forge a client identity.
- **Identity per VM.** Each enrolled VM has its own signed
  identity cert, CN = VM UUID. The PKCS#12 bag is embedded in
  the enrollment profile; `mdmclient` extracts it on first
  install and signs every subsequent message with the
  matching private key.

## What's intentionally out of scope

- **No APNs.** `mdmclient` polls instead. Sufficient for
  host-controlled VMs; saves the operator from running a
  push-cert renewal pipeline.
- **No Apple Business Manager.** VMs aren't DEP-registered;
  the enrollment profile is installed via the in-bundle
  bootstrap rather than pushed by Apple.
- **No declarative management.** Only the four command kinds
  (`InstallEnterpriseApplication`, `InstallProfile`,
  `RemoveProfile`, plus the implicit check-in messages) are
  modeled. Declarative configurations are a separate Apple
  schema and aren't needed for host-driven runner / desktop
  workloads.
- **No fleet-wide remote wipe.** `mdmclient` supports
  `EraseDevice` etc., but those commands are deliberately
  not exposed in `MDMCommand.Kind`. Host owns the VM disk
  directly; if you need to destroy a VM, use `spook delete`.

## Simplification log

The embedded MDM was built in seven internal phases (DTOs →
identity issuer → HTTP server → command DTOs → polling →
VM-create injection → user-data delivery). Once all phases
shipped, the public docstrings still pointed forward at
"Phase 2's CA work lands", "Phase 7 ships the unsigned pkg",
etc. — references that were useful during the build but had
rotted into noise.

The cleanup pass (PRs ending in `de07cbd5` + `ee11dda5`)
removed:

| Category | What | Why |
|---|---|---|
| Dead API | `MDMContentStore.register()` | Unused in production. The dispatcher mints its own UUID up front so it can bake it into the manifest URL before storing — meaning the convenience wrapper that auto-generates the UUID is never called by `MDMUserDataDispatcher`. The five test call sites were migrated to `store(id:)`. |
| Indirection | `MDMUserDataPkgBuilding.BuiltPackage` typealias | One name for two surfaces. The typealias re-exported `MDMUserDataBuiltPackage` inside the protocol scope; every caller already used the full name. Collapsed to a single name. |
| Stale docs | "Phase N" / "Phase 2's CA work lands" / "see Phase 5" | Internal build-phase pointers tell future readers about the temporal order of the build, not the present state of the code. Rewritten in present tense. |
| Factual bug | CLI `Devices` subcommand: "Until persistence ships, this command can only see devices that enrolled during the currently-running `serve`" | Persistence shipped via `MDMDeviceStorePersister.flush()`, called on every mutation. The doc was lying. |
| Vestigial | `DiskInjector.generateLaunchDaemonPlist()` + `daemonLabel` + `guestScriptPath` + `xmlEscape` + `chownFailed` error case | Leftover from when `DiskInjector` wrote a LaunchDaemon plist directly into the guest filesystem. Once Guest Tools' `Spooktacular Provisioner.pkg` started owning the runner daemon and watching the virtio-fs `provision/` share for `first-boot.sh`, the host-side plist generator stopped having any callers. The associated `chownFailed` error case (from the also-ripped `osascript`+`chown` step) became unreachable. |
| Trailing pointer comments | `// MDMCommand lives in MDMCommand.swift` etc. | The compiler already knows where types live. Pointer comments are stale on the first rename. |

**Why these specific principles drive the cleanup**

1. **Two names for one thing is harder to read than one.**
   `BuiltPackage` and `MDMUserDataBuiltPackage` were the same
   type; the indirection earned nothing.
2. **Phase-N references are version-control history written
   into source code.** The git log already preserves the
   build order; the source itself should describe the
   present.
3. **Dead APIs aren't free.** Every reader has to decide
   which of two near-identical methods to use, and tests
   drift toward the convenience overload while production
   drifts toward the explicit one — exactly the bifurcation
   that bit the content-store API here.
4. **Vestigial error cases hide the live ones.** Future
   readers spend time reasoning about how `chownFailed`
   could fire before realising it never does.

After the cleanup, the embedded MDM module is **170 fewer
lines of code** at 1057/1057 Swift Testing tests passing.
