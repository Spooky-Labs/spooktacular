01 · CI/CD

Ephemeral GitHub runners on your Mac fleet.

One VM, one job, destroyed. The strictest isolation guarantee possible on macOS — every job runs on a byte-for-byte fresh install. Warm-pool mode swaps the destroy-and-re-clone cycle for a validated scrub when cold-start cost is the bottleneck.

The shape of a pool

A RunnerPool CRD declares mode, min/max, and source VM. The operator does the rest: clone, register, drain when done, recycle or destroy. Scale to zero, scale on queue depth, or pin min capacity.

  • Ephemeral mode — one job per VM, clean destroy-and-reclone
  • Warm-pool mode — 10-check scrub validator proves clean
  • Persistent mode — long-lived runners with drain-on-shutdown
  • Keychain tokens — registration tokens never in ps, env, or disk
  • Workload JWTs — JWKS kid rotation with 24 h overlap window
  • Drain-before-delete — no torn builds on scale-down
runners.yaml 
apiVersion: spooktacular.io/v1
kind: RunnerPool
metadata:
  name: ios-ci
  namespace: mobile-platform
spec:
  mode: ephemeral
  minSize: 2
  maxSize: 8
  sourceVM: macos-26-xcode-16
  github:
    repo: acme/platform-ios
    tokenKeychainAccount: ios-ci-pool
  scrub:
    validator: clean-tenfold
    timeout: 90s
Per-label shell escape. Labels feed into the runner template as a single --labels argument. The template POSIX-escapes every label individually before joining — a single-quote inside a label can't break out and inject commands. Catches the classic "malicious repo name in label" vector.
02 · Fleet provisioning

Re-image the fleet overnight. Without a network round-trip.

Create one golden VM. Clone it by the dozen. Every clone gets a fresh machine identifier, a unique MAC, and an injected user-data script that runs on first boot via a LaunchDaemon — no guest agent, no SSH, no DHCP required.

Bulk provisioning from a single base

APFS copy-on-write means a 100 GB disk clones in milliseconds and shares blocks until the clone writes. A 32-host rack with 2 VMs per host fits 64 VMs cloned from one IPSW install. All re-imageable on demand.

  • disk-inject — LaunchDaemon dropped pre-boot, runs before Setup Assistant
  • Setup Assistant automation for macOS 15 and 26 — unattended
  • Stable MAC addresses — IP resolution works on first boot, no manual DHCP reservation
  • Bundle.config.json — atomic writes, versioned, git-friendly
  • Snapshot → restore per VM — reset to known-good between runs
  • Shared folders for config, certs, and scripts — no secrets on the guest image
fleet-reset.sh
# Snapshot the golden state once
$ spook snapshot save macos-26-base golden
✓ Snapshot 'golden' saved (142 MiB aux).
 
# Roll out 24 identical VMs
$ for i in {01..24}; do
    spook clone macos-26-base host-$i \
      --user-data ./provision-$i.sh
done
 
# Morning reset — snapshot-restore in parallel
$ spook snapshot restore host-* golden
✓ 24 VMs reset in 2.1s.
03 · Virtual Desktop Infrastructure

Persistent Macs for remote developers, contractors, and BYOD users.

Give your team a real Mac desktop hosted on centralized infrastructure. Metal-accelerated display, auto-resize, audio, clipboard, shared folders. Screen Sharing enabled on first boot — connect with any VNC client.

Display

1 or 2 virtual displays, Metal GPU, auto-resize picks up window-size changes so guest resolution tracks the client — essential for remote desktop.

Audio + mic

VirtIO sound devices. Toggle per-VM from the Hardware editor. Disable cleanly for headless compute — removes the device, doesn't just mute.

Shared folders

VirtIO FS mounts under /Volumes/My Shared Files/. Pass build artifacts, credentials, training data without touching the network. Read-only per-folder.

Clipboard bridge

Host ↔ guest pasteboard sync for Linux guests via the SwiftUI app. macOS guests use the native Apple clipboard bridge when available.

Remote Desktop template

One flag (--remote-desktop) enables macOS Screen Sharing on first boot and surfaces the VNC URL in the system log. Any VNC client works.

Workspace windows

Each running VM opens in its own SwiftUI window. Close the library, keep the workspaces — VMs feel like apps on the Dock.

Provision once, recycle forever. Pair a long-lived VDI base with spook snapshot restore for "roll back the broken install" — users get their environment back in seconds without IT ticket churn.
04 · Computer-using agents

A sandboxed macOS desktop for every agent.

Give a computer-using agent its own Mac — Finder, Safari, Xcode, every app the host Mac can run. The Spooktacular guest agent exposes what the agent actually needs (exec, file I/O, clipboard, app launch, port listing) over VirtIO sockets, separated into three trust tiers so the agent's session scope matches its permissions.

Why this is not a container

Containers share the host kernel. An agent that runs a full macOS browser, exercises Xcode's code signing, or hits a Keychain simply can't do that inside a Linux container — you need a real macOS kernel. Spooktacular gives every agent one, isolated by Apple's Virtualization.framework.

  • OpenClaw template — Node.js + gateway daemon, ready for agent frameworks
  • Secrets via shared folder — API keys never in env, never in the script
  • Three-tier vsock — read-only / runner / break-glass ports
  • Egress allowlist compiled to pf(8) rules — agent can't dial the open internet
  • Per-VM snapshot to roll back after each agent session
  • Ephemeral mode for one-shot tasks — VM destroyed after run
agent-host.sh
# Create a sandbox with the OpenClaw template
$ spook create agent-01 \
    --openclaw \
    --share ~/agent-secrets \
    --network isolated
 
# Talk to the guest agent over vsock — runner port
$ spook exec agent-01 -- \
    "open -a Safari https://example.com"
 
# Roll back after the session
$ spook snapshot restore agent-01 clean
✓ Rolled back to snapshot 'clean'.
Break-glass with a signed ticket. If an agent needs privileged admin access, the operator issues an Ed25519-signed ticket bound to the VM's fleet and a specific JTI. The ticket is single-use (UsedTicketCache is distributed) and replay-proof — an agent can't leak it twice.
